# PIPELINE — 从一句话需求到能跑的代码(体量自适应流水线)

> **定位**:空项目开局的导航层。协作协议(PROTOCOL.md + 三环)解决"两个 AI 怎么协作",本流水线解决"协作着**做什么、按什么顺序做**"。
> **血统**:蒸馏自 OPC 项目数月实战(PRFAQ→PRD→BRD→TAD→POC→spec→代码,数百轮双 AI 审稿)。**搬的是机制,不搬吨位**——每个机制后面标了它防的事故。
> **铁律**:上游文档是下游的宪法。需求书锁"做什么",规划书锁"怎么搭",模块规约锁"这块怎么写",代码不许绕规约;改上游 = 硬升级,必须 Owner 拍板。
> **前置**:项目已完成部署(START.md 自举或 QUICKSTART 安装,`_collab/` 总线存在)——流水线的交审、账本都跑在总线上,没部署先部署。

---

## 一、总览:四阶段 + 门禁

```
Owner:"我要做一个 XXX"(一句话)
   │
阶段0 需求 ─ 开发员起草需求书 → 审核员对抗审 → Owner 逐条拍板   ◄ 门禁①(含体量定档)
阶段1 规划 ─ 开发员起草规划书 → 审 → Owner 拍板                ◄ 门禁②(轻型并入门禁①)
阶段2 规约 ─ 逐模块 unit spec → 审 → Owner 说"开始"            ◄ 门禁③(= lean 硬停点①)
阶段3 执行 ─ lean 增量循环(写→审→过→下一增量)→ Owner 亲测    ◄ 门禁④(= lean 硬停点②)
   │
下一模块回到阶段 2,直到收工
```

- 除议事 detour 外全程 `mode = execution --lean`(Owner 直驱,审核员按 READY 信号审),不折腾模式切换
- 文档也是交付单元:需求书/规划书各按**审稿环**走一轮(unit_id 如 `requirements` / `plan`)
- 阶段 0/1 中拿不准的**重大取舍** → 转**议事环**盲发双观点再回来(见 §四)
- **每个门禁的拍板都进决策账本**(见 §五)——这是全流水线的锚

## 二、体量三档(门禁①一并拍板,决定后面走多重)

开发员在需求书"体量评估"节按下表给出建议档位 + 依据,Owner 拍板定档:

| 信号 | **轻型** | **标准** | **重型** |
|---|---|---|---|
| 预期模块数 | 1~2 | 3~8 | 9+ |
| 数据 | 无持久化 / 单文件 | 数据库 / 多实体 | 多实体 + 状态机 + 迁移 |
| 外部集成 | 0~1 个 | 2~3 个 | 多方集成 / 多端 |
| 安全敏感 | 无凭证 | 有凭证或用户数据 | 凭证 + 权限体系 + 审计 |
| 工期直觉 | 数天 | 数周 | 数月 |

**各档产出物**(信号跨档取高不取低;拿不准 → 取高一档):

| | 轻型 | 标准 | 重型 |
|---|---|---|---|
| 阶段0 | 需求书**薄芯**(7 节,★§7 决策清单不可省;轻型 §6 含"迷你规划") | 需求书薄芯 | 需求书薄芯 + 扩展节 |
| 阶段1 | **跳过**(迷你规划并入需求书,门禁②并入①) | 规划书标准芯 | 规划书标准芯 + 重型节(实体/状态机锁定、契约约定)+ **风险探针先行** |
| 阶段2 | 可能全项目就 1 份 spec | 每模块 1 份 spec | 同标准,spec 须引用规划书锁定的实体/状态机 |
| 门禁数 | 3(①③④) | 4 | 4 + 探针放行 |

**中途升档**:执行中发现实际比定档重(模块膨胀、数据复杂化、突然要接第三方)——这本身就是范围信号,**停下告诉 Owner**,Owner 同意后按新档补课再继续:**轻型→标准 = 从零起草完整规划书,走阶段 1 全流程(审稿环 + 补过门禁②)**——未审未拍板的规划书直接生效等于绕门禁;标准→重型 = 补规划书重型节(实体/状态机锁定等),同样走一轮审 + Owner 拍板。升档决定本身入账本。降档同理但少见。

## 三、四阶段细则

### 阶段 0:需求(模板 [templates/requirements.md](templates/requirements.md))

- **谁做**:开发员起草;**你只需要给一句话 + 回答它的提问**
- **存放约定**:模板拷为项目根 `docs/requirements.md`(规划书同理 `docs/plan.md`;项目想放别处也行,但路径在门禁①随需求书锁定入账,两个 AI 从此都从这里找)
- **灵魂段是"Owner 决策问题清单"**:凡是 AI 不该替你定的取舍(要不要登录?单机还是联网?免费还是收费?),它必须列成问题——每条带选项、各自代价、AI 推荐——而不是替你写成结论。**AI 写需求最大的危险就是顺手替你把产品做了主**(OPC 铁律"两 AI 共识 ≠ Owner 决策"在需求阶段的投影)
- **"明确不做"节和"要什么"同等重要**:诚实 scoping,防范围蔓延(OPC 的 MVP 边界纪律)
- 走一轮审稿环(审核员专挑:漏掉的场景、自相矛盾、验收标准不可测)→ 门禁①:你逐条回答决策清单 + 定档 → 需求锁定,拍板进账本
- **定档与建议不符时**(比如开发员建议标准、你拍轻型):开发员按新档增删对应节(如补迷你规划),增补部分由审核员快速复核一轮后再锁定——不带着没审过的段落进锁定态

### 阶段 1:规划(模板 [templates/plan.md](templates/plan.md);轻型跳过)

- 开发员起草:架构一图流、技术选型(每项带"被否选项+一句话理由")、模块切分、**建造序**(按依赖排,先地基后上层——OPC 建造序机制)、风险探针清单
- **决策清单的反向过滤器(OPC ADR 教训)**:技术实施细节(算法、库、内部 schema、索引、代码组织)**不进 Owner 决策清单**——AI 在选型表自决并留"被否选项+理由"痕迹;只有命中硬升级类别或产品级取舍(付费/隐私/范围/安全体位)才列给 Owner。OPC 实战反例:曾把 30 项决策堆给 Owner,其中 22 项本属技术自决——"什么都问"和"什么都不问"同样是失职,Owner 注意力是最稀缺资源
- **探针时点**:重型档探针**先行强制**(跑完探针放行门才进阶段 2);标准档只须列出清单,哪些要先验由你在门禁②裁量
- **重型附加**:核心实体表 + 状态机锁定(OPC BRD 02/03 章的精简版——这两样一旦开码再改,成本是灾难级的,所以重型必须在此锁死);**高风险技术假设先做探针验证再进阶段 2**(OPC POC 机制:探针是验证风险,不是范围扩张,探针代码不直接沉淀为正式架构)
- 走审稿环 → 门禁②:你拍板 → 规划锁定入账本

### 阶段 2:模块规约(套件已有:[../templates/unit_spec.md](../templates/unit_spec.md))

按建造序逐模块写 spec(9 节)→ 审 → **你说"开始"才动代码**(门禁③)。spec 的"现状核实"节必须亲读已有代码;**凡有规划书的档位(标准+重型),spec 引用到的实体、模块职责、已锁决策必须逐条给出处(规划书节号 / 账本 D-NNN)并声明已亲核**,审核员对照打勾——OPC 血泪:一次模块名引用凭印象写错,沿链路传染了一整串文档,连审核员都没抓住;引用必核对上游权威,不凭记忆。

### 阶段 3:执行(套件已有:[../loops/lean-code-mode.md](../loops/lean-code-mode.md))

lean 六步增量循环 + dev-log 随码交付 + 增量间自动续跑;模块完成**你亲测**(门禁④)——AI 的"全绿"只证明 AI 想得到的都验了,你亲测验的是它想不到的(OPC 实证:三层审查全绿后 Owner 亲测第一屏就发现门坏了)。

## 四、什么时候转议事环

阶段 0/1 里出现**有真实代价、你确实拿不准、两个 AI 可能有不同看法**的取舍(如技术栈二选一、本地 vs 云端)→ 值得为它单开一个议事(`topic` 命令,轻量档 `--lite` 通常够用),拿到盲发双观点 + 5 行摘要再拍板。琐碎取舍别开议事,直接在决策清单里勾选——**议事环留给"拍错疼半年"的问题**。

**detour 的回程**:议事收口(`decision` 命令)后 mode 会落在 idle;回到流水线时,你给下一个任务的同时,被派活方按 PROTOCOL §四"开工授权例外"重跑 `collab.mjs mode execution --lean` 恢复(decision 命令的输出里也会提醒这一句)。

## 五、决策账本(`_collab/locked_decisions.md`)

- **每个门禁的拍板、每次议事的采纳、执行中的每次硬升级裁断**,都追加一条 D-NNN 进账本(一句话决策 + 一句话理由 + 出处)
- 誊录走脚本(机械,AI 不得擅自增删改):

```bash
node ai-collab-kit/scripts/collab.mjs decide "素材只存本地,MVP 不做云同步" --why "隐私+成本" --source "门禁①"
```

- **入账粒度**:按决策逐条入账——门禁①的清单有 5 个 Q 就入 5 条 D-NNN,定档本身也单入一条;誊录靠人跑命令(**不自动**,漏誊录 = 该门禁交付不完整)
- **账本是硬升级第 1 类的判定依据**:此后任何文档、任何代码与账本冲突 → 立即 BLOCKED 找你,两个 AI 意见再一致也不例外。你在阶段 0 拍的板,到阶段 3 第 50 个增量时依然有效——这是 OPC"已锁口径 + Owner 拍板追踪"机制(已锁决策清单 + 决策登记 + 拍板节点跟踪表)的通用化,它是整条流水线不随时间漂移的原因

## 六、常见问题

- **需求中途要变怎么办?** 正常且欢迎——但走正门:变更 = 改上游宪法 = 硬升级,告诉任一 AI"我要把需求 X 改成 Y",它会停下走 Owner 拍板 → 更新需求书 + 账本记录(旧决策标废弃,不删——历史要可追溯)→ 受影响的下游文档同步修
- **很小的工具项目也要走流水线?** 轻型档的文档仪式 = 一页需求书(含迷你规划)+ 一份 spec;你出现 3 次(定档、说"开始"、亲测)。连这个都想省的项目,直接 lean 裸奔也行——流水线是导航不是牢笼;但只要项目会活过一周,那一页需求书就会开始还债
- **文档写多细?** 每层只锁自己该锁的:需求书不写技术方案,规划书不写代码细节(OPC 层级纪律:字段级细节写进需求书 = 越层,审核员应打回)
