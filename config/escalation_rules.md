# 硬升级清单(escalation_rules)——本项目"AI 不许拍板"的边界

> **这是整个协议里最重要的一份配置。** 命中下列任一类的议题/发现/变更,**即使 Writer 和 Reviewer 意见完全一致,也必须停下交 Owner 拍板**(审稿环直接 `BLOCKED + owner_decision_required_<topic>`;议事环照常讨论但强制 `escalated_owner`)。
>
> 原理:两 AI 共识只能证明"按当前材料看内部一致",不能证明 Owner 决策背后的产品意图、成本约束、政策风险、用户心智已失效。在这些维度上两个 AI 共同失明——一致只说明盲区重叠。
>
> **使用方式**:init 后本文件在 `_collab/escalation_rules.md`,由 Owner 按项目增删改。改本清单本身也是硬升级(第 1 类)。

---

## 通用默认五类(按项目裁剪)

### 1. 推翻任何 Owner 已拍板的决策

- 本项目"已锁决策"记录在:`<填:决策记录位置,如 docs/decisions/ 或某文档的决策章节>`
- 包括:推翻已锁口径、绕过已锁方案另起炉灶、"优化"已拍板的取舍
- AI 可以做的:指出已锁决策与新事实的冲突 + 给利弊分析 → 交 Owner

### 2. 数据契约权威变更

- 核心实体的性质/生命周期语义变更(不是加字段,是**改性质**)
- 状态机权威变更(状态增删、迁移规则改动)
- 对外 API / 存储 schema 的破坏性变更
- 本项目的契约权威文档:`<填:如 docs/data-model.md>`

### 3. 范围边界移动

- MVP ↔ 后续版本之间搬功能(任一方向)
- scope 增删(加需求、砍需求)
- "顺手做了"型范围扩张(最隐蔽的一种——以"反正代码都打开了"为理由)

### 4. 安全信任边界变更

- 凭证/密钥/token 的存储、注入、流转方式
- 权限模型、沙箱策略、审计归属
- 信任分层变更(什么代码可以接触什么敏感数据)
- **注意**:这一类连"两个 AI 都觉得更安全"的改动也要升级——安全体位是 Owner 的风险决策

### 5. 不可逆或对外动作

- 发版 / 部署到生产 / 对外发布内容
- 删数据、改历史(git push --force、改写已发布记录)
- 花钱(开通付费服务、扩容)
- 给第三方发消息/邮件/开 issue

## 判定纪律

- 议事产物末尾的 `hard_escalation_category` 字段必须从本清单的类别名中取值
- **拿不准算不算命中 → 按命中处理**(宁可多问一次 Owner,不可替 Owner 拍一次板)
- 实现"Owner 已拍板"的决策不算命中(那是执行不是决策)——但交审时必须带"Owner 已拍板"横幅,见 `loops/lean-code-mode.md` §五
- **本文件与 `collab.config.json` 由 Owner 亲自维护**:提交时用前缀 `owner(config): ... [skip-review]`(机检只放行此前缀触碰这两个文件);AI 使用此前缀即越权,会在 git log 里显形
